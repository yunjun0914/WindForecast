# 외부데이터 (KMA AWS/ASOS 관측)

작성일: 2026-07-09

대회 규칙(docs/rules.md)상 외부 공개 데이터는 허용되며, 관측자료는 관측 직후 준실시간 공개되므로 각 예측값의 예측기준시점(전일 ~13:00) 이전 자료만 쓰면 leakage 규칙을 준수한다.

## 1. 데이터 위치와 형태

| 항목 | 내용 |
|---|---|
| 경로 | `data/external/kma_aws_hourly/aws_YYYYMM.csv.gz` (월별) |
| 수집 완료 | 2022-01 ~ 2024-03 (27개월). 2024-04~12는 일일 API 한도로 미수집, 재실행 시 이어받음 |
| 크기 | 월 ~6.5MB, 총 ~175MB |
| 행 | 시간별 x 전 지점(~727개, AWS + ASOS 포함 — ID<300이 ASOS) |
| 컬럼 | `tm`(KST 시각), `stn`(지점ID), `ta`(기온), `wd`(풍향), `ws`(풍속), `rn_day`/`rn_hr1`(강수), `hm`(습도), `pa`/`ps`(기압) |
| 결측 코드 | 음수(-9, -99 등) -> NaN 처리 필요 |

## 2. 출처/수집 (2차 평가 소명용)

| 항목 | 내용 |
|---|---|
| 출처 | 기상청 API허브 `https://apihub.kma.go.kr/api/typ01/url/awsh.php` (방재기상관측 AWS 시간자료) |
| 수집 스크립트 | `data/external/collect_kma_aws_hourly.py` |
| 인증키 | 환경변수 `KMA_API_KEY` (코드에 하드코딩 안 함) |
| 수집 시점 | 2026-07-09 |
| 재현 | `$env:KMA_API_KEY="..."; python data/external/collect_kma_aws_hourly.py --start 2022-01-01 --end 2025-01-01` |
| 공개성 | 누구나 API허브 가입 후 접근 가능한 공개 데이터 |
| leakage | 2022~2024 자료는 모든 2025 예측기준시점 이전에 공개 완료. 2025 관측을 test 피처로 쓸 경우 각 행의 예측기준시점 이전 시각 자료만 사용해야 함 |
| 주의 | 일일 호출 한도 존재(~2만콜/일 수준). 시각당 1콜(전 지점 일괄)이라 1년치 = 8,760콜 |

## 3. 검증 결과 요약

### 3-1. 지점별 단순 상관 (하한) — `experiments/audit_kma_aws_bridge.py`

- 최고 단일 지점(596/595/100=대관령) corr 0.65~0.72, 최근접 ASOS 태백(216, 분지)은 0.27~0.35로 최악.
- top5 단순 평균 0.78~0.82 = 기존 예보 gfs_ws850(0.81)과 동급.
- 이 결과만 보고 처음엔 no-go 판단했으나 **이는 하한 측정이었음 (판단 오류로 정정됨)**.

### 3-2. 다변량 모델 A/B/C (핵심 결과) — `experiments/evaluate_aws_teacher_value.py`

학습 2022~2023, 검증 2024 Q1. target = group SCADA 현장풍속.

| 모델 | g1 corr | g2 corr | g3 corr | MAE(g1/g2/g3) |
|---|---:|---:|---:|---|
| A: 예보 피처만 (85) | 0.9141 | 0.9090 | 0.9002 | 1.095 / 1.235 / 1.280 |
| B: 관측 636지점만 | 0.9103 | 0.9134 | 0.9068 | 1.114 / 1.235 / 1.270 |
| C: 예보 + 관측 | **0.9294** | **0.9300** | **0.9245** | **1.013 / 1.090 / 1.128** |

결론:

- 관측망 다변량 조합(B)은 예보(A)와 동급 — 계곡 지점들로도 능선 바람 삼각측량 가능.
- **C > A 전 그룹 일관 (+0.015~0.024 corr, MAE -7~-12%) — 관측망에는 예보에 없는 현장풍속 정보가 실재.**
- 결과 파일: `results/aws_teacher_value_ab.csv`, `results/kma_aws_bridge_station_corr.csv`

### 3-3. 모델 D: 합법 lagged 관측 피처 — `experiments/evaluate_aws_lagged_teacher_value.py`

목적: C는 같은 시각 관측을 쓰므로 test에 직접 사용할 수 없다. 따라서 각 row의 `data_available_kst_dtm` 이전 AWS 관측만 사용해도 예보 단독(A)을 넘는지 확인했다.

학습/검증은 A/B/C와 동일하게 2022~2023 학습, 2024 Q1 검증. target = group SCADA 현장풍속.

| 모델 | 설명 | g1 corr | g2 corr | g3 corr | MAE(g1/g2/g3) |
|---|---|---:|---:|---:|---|
| A: 예보 피처만 | LDAPS/GFS + meteo | 0.9141 | 0.9090 | 0.9002 | 1.095 / 1.235 / 1.280 |
| D_base_obs | A + `data_available_kst_dtm` 시각 전지점 AWS ws | 0.9076 | 0.8940 | 0.8909 | 1.130 / 1.309 / 1.321 |
| D_base_roll6 | A + base AWS ws + 6h rolling mean + 6h delta | 0.9076 | 0.8958 | 0.8924 | 1.117 / 1.313 / 1.326 |
| C upper-bound | A + 예측 대상 시각의 AWS ws (test 직접 사용 불가) | 0.9294 | 0.9300 | 0.9245 | 1.013 / 1.090 / 1.128 |

판단:

- **D는 A보다 전 그룹에서 악화.** 단순 base-time lagged 관측은 12~35시간 뒤 현장풍속을 보정하는 데 직접 이득이 없다.
- C upper-bound는 여전히 강하므로 관측망 정보 자체는 유효하지만, 이를 실전 피처로 쓰려면 raw lag value가 아니라 **예보-관측 bias, issue-time 보정, 또는 privileged teacher distillation** 형태가 필요하다.
- 결과 파일: `results/aws_lagged_teacher_value_d.csv`

## 4. 다음 단계

현재까지 확인한 점:

| 형태 | 결과 |
|---|---|
| D: base-time lagged 관측 직접 | A보다 악화 |
| TREE naive 전 그룹 투입 | 전체 평균 악화, group2는 양수 |
| group2-only selected | Q1 quick valid에서 양수 |
| group별 importance top10 투입 | group2/group3 양수, group1 악화 |
| group2/group3 top10 부분 적용 | 2024 전체 기준 평균 소폭 양수, group3 중심 |
| same-time C | 정보량 실재하나 test 사용 불가 |

다음 후보:

1. same-time AWS는 final/test feature로 절대 사용하지 않는다. upper-bound 진단용으로만 둔다.
2. final 후보는 `data_available_kst_dtm` 이하 관측만 사용한다.
3. station metadata를 이용해 group별 station selection을 다시 설계한다.
4. group1은 현재 top10 방식이 악화했으므로 별도 selection 기준이 필요하다.
5. group2/group3는 AWS 신호가 일부 보였으므로, feature 축소/권역화/PCA/변수 확장(`wd`, `pa`, `ps`, `ta`, `hm`)을 검토한다.
6. privileged distillation(C를 train-only teacher로, 추론은 예보만)은 별도 후보로 보류한다.
